# 考纲：考试科目1：系统分析师综合知识

## 1. 计算机系统基础知识

### 1.1 计算机系统概述

1.1.1 计算机系统层次结构

	● 硬件层、系统层、应用层
	硬件层（裸机）：该层可细分为硬联逻辑级、微程序级和传统机器级。
	系统层：该层课细分为操作系统级、语言处理程序级（如编译程序、汇编程序和解释程序）。
	应用层：该层是针对不同用户而设计的各种应用程序。

1.1.2 计算机系统硬件组成

	● 冯·诺依曼计算机结构
	运算器、控制器、存储器、输入设备和输出设备。
	中央处理单元CPU包括运算器和控制器。

1.1.3 计算机软件

	● 系统软件、应用软件
	系统软件：用于实现计算机系统的管理、调度、监视和服务等功能。通常将系统软件分为 5 类：操作系统、语言处理程序、服务性程序、数据库管理系统和计算机网络软件。
	应用软件：是为解决某种应用问题而编制的一些程序。应用软件是用户或第三方软件公司为各自业务开发和使用的各种软件。例如，财务管理软件、项目管理软件、营销管理软件等。

### 1.2存储器系统

1.2.1 主存储器

	● RAM、ROM
	RAM, Random Access Memory：随机存取存储器，既可以写入也可以读出，但断电后信息无法保存，因此只能用于暂存数据。
	ROM, Read Only Memory：只读存取存储器，只能随机读出而不能写入，即使断电后信息也不会丢失。ROM 一般用于存放系统程序BIOS (Basic Input Output System， 基本输入输出系统）、专用的子程序，或用作函数发生器、字符发生器及微程序控制器中的控制存储器。

1.2.2 辅助存储器

	● 磁带存储器
	磁带存储器由磁带机及其控制器组成，属于顺序存取设备。
	● 硬盘存储器
	硬盘存储器简称硬盘，属于直接存取设备。硬盘分为三类：机械硬盘 HDD、固态硬盘 SSD 和混合硬盘 SSHD。
	机械硬盘是传统硬盘，由磁性盘片、驱动系统、读写系统和控制系统组成。
	固态硬盘是由多个闪存芯片加主控以及缓存组成的阵列式存储设备，属于以固态电子存储芯片阵列制成的硬盘。
	混合硬盘是机械硬盘与固态硬盘的结合体，采用容量较小的闪存用来存储常用文件，而磁盘才是最主要的存储介质，闪存仅起到缓冲作用。
	● 磁盘阵列
	RAID 现在代表独立磁盘冗余阵列（Redundant Array of Independent Disks）。RAID 应用的主要技术有分块技术、交叉技术和重聚技术。RAID 分为以下 8 个级别：
	RAID 0、RAID 1、RAID 2、RAID 3、RAID 4、RAID 5、RAID 6、RAID 7、RAID 10。
	● 光盘存储器
	光盘存储器是利用激光束在记录表面存储信息，根据激光束的反射光来读出信息。光盘存储器可以分为4种类型：CD-ROM、CD-R、CD-RW、DVD-ROM

1.2.3 高速缓冲存储器

	● Cache 基本原理
	使用 Cache 改善系统性能的依据是程序的局部性原理。程序访问的局部性有两个方面的含义，分别是时间局部性和空间局部性。时间局部性是指如果一个存储单元被访问，则可能该单元会很快被再次访问；空间局部性是指如果一个存储单元被访问，则该单元邻近的单元也可能很快被访问。
	● 映射机制
	当 CPU 发出访问存储请求后，存储器地址先被送到 Cache 控制器以确定所需数据是否已在 Cache 中，若命中，则直接对 Cache 进行访问。这个过程称为 Cache 的地址映射（映像）。常见的映射方法有直接映射、全相联映射、组相联映射。
	● 替换算法
	当 Cache 已经存满数据后，新数据必须替换 Cache 中的某些旧数据。最常用的替换算法有随机算法、先进先出算法和近期最少使用算法。
	● 写操作
	需要保证缓存在 Cache 中的数据与内存中的内容一致。常用以下集中方法：写直达（写 Cache 时同时写回内存）、写回（Cache 失效时写回内存）、标记法（写入内存同时标记 Cache 失效）。

1.2.4 网络存储技术

	● 直接附加存储 DAS
	DAS, Direct Attached Storage：直接附加存储不带有任何存储操作系统，通过 SCSI 电缆直接连到服务器。
	● 网络附加存储 NAS
	NAS, Network Attached Storage：网络附加存储类似于一个专用的文件服务器，去掉了通用服务器的大多数计算功能，仅提供文件系统功能。通过网络接口与网络直接相连，用户通过网络访问。
	● 存储区域网络 SAN
	SAN, Storage Area Network：通过专用交换机将磁盘阵列与服务器链接起来，没有采用文件共享存取方式，而是采用块级别存储。

1.2.5 虚拟存储技术

	● 虚拟存储的分类
	按照拓扑结构的不同，虚拟存储可以分为对称式和非对称式两种。
	对称式是指虚拟存储控制设备与存储软件系统、交换设备集成为一个整体，内嵌在网络数据传输路径中；
	非对称式是指虚拟存储控制设备独立于数据传输路径之外。
	● 虚拟存储的实现方式
	虚拟存储要解决的关键问题是逻辑卷与物理存储实体之间的映射关系，这种映射关系可以在计算机层解决，也可以在存储设备层解决，还可以在存储网络层解决。
	因此分为：主机级虚拟化、存储设备级虚拟化、网络级虚拟化。
	● 虚拟存储的特点
	虚拟存储技术是为了解决复杂、烦琐的存储管理而产生的，广泛应用于数据镜像、数据复制、实时数据恢复、应用集成等方面。
	（1）虚拟存储提供了一个大容量存储系统集中管理的手段，由网络中的一个环节进行统一管理，避免了由于存储设备扩充所带来的管理方面的麻烦。
	（2）虚拟存储可以大大提高存储系统整体访问带宽。很好地进行负载平衡，把每一次数据访问所需的带宽合理地分配到各个存储模块上。
	（3）虚拟存储技术为存储资源管理提供了更好的灵活性和兼容性，可以将不同类型的存储设备集中管理使用。
	（4）虚拟存储技术可以通过管理软件，为网络系统提供一些其他的有用功能。
	（5）虚拟存储技术将计算机的应用系统与存储设备分离，使各种不同的存储设备具有标准的存储特性，而应用系统无须关心数据存储的具体设备，减轻了应用系统的负担。

### 1.3 输入输出系统

1.3.1 输入输出工作方式

	I/O 系统可以有 5 种不同的工作方式，分别是程序控制方式、程序中断方式、DMA 工作方式、通道方式、I/O 处理机。
	● 程序控制方式
	程序控制方式是由CPU 执行一段 I/O 程序来实现主机与外设之间的数据传送，根据外设的不同性质，可分为无条件传送和程序查询方式两种。
	在无条件传送方式中，I/O 端口总是准备好接收主机的输出数据，或总是准备好向主机输入数据，因而 CPU 无须查询外设的工作状态，而默认外设始终处于准备就绪状态。
	在程序查询方式种，为了保证数据传送的正确进行，就要求 CPU 在程序中查询外设的工作状态。如果外设尚未准备就绪，CPU 就循环等待，只有当外设己做好准备时，CPU 才能执行 I/O 指令进行数据传送。
	● 程序中断方式
	程序中断是指计算机执行现行程序的过程中，出现某些急需处理的异常情况或特殊请求，CPU 暂时中止现行程序（保护现场），而转去对随机发生的更紧迫的事件进行处理，在处理完毕后，CPU 将自动返回原来的程序继续执行（恢复现场）。
	整个中断过程大体上可以分为 5 个阶段，分别是中断请求、中断判优、中断响应、中断处理和中断返回。
	● DMA工作方式
	DMA 工作方式是为了在主存与外设之间实现高速、批量数据交换而设置的，使用 DMA 控制器（Direct Memory Access Controller,DMAC）来控制和管理数据传输。
	DMAC 和 CPU 共享系统总线，并且具有独立访问存储器的能力。在进行 DMA 时，CPU 放弃对系统总线的控制而由 DMAC 控制总线，由 DMAC 提供存储器地址及必需的读写控制信号，实现外设与存储器之间的数据交换。
	● 通道方式
	通道是一种高级的 I/O 控制部件，它在一定的硬件基础上利用软件手段实现对 I/O 的控制和传送，更多地免去了 CPU 的介入，从而使主机和外设的并行工作程度更高。
	● I/O 处理机
	I/O 处理机也称为外围处理机，具有丰富的指令系统和完善的中断系统。专用于大型、高效的计算机系统处理外围设备的 I/O , 并利用共享存储器或其他共享手段与主机交换信息，从而使大型计算机系统更加高效地工作。

1.3.2 总线

	● 总线的分类、标准和性能指标
	总线是一组能为多个部件分时共享的公共信息传送线路。
	共享是指总线上可以挂接多个部件，各个部件之间相互交换的信息都可以通过这组公共线路传送；
	分时是指同一时刻只允许有一个部件向总线发送信息，若出现两个或两个以上部件同时向总线发送信息，则会导致信号冲突。但在同一时刻，允许多个部件同时从总线上接收相同的信息。
	
	按总线相对于 CPU 或其他芯片的位置，可以分为内部总线和外部总线两种。在 CPU 内部，寄存器之间和算术逻辑部件（ALU）与控制部件之间传输数据所用的总线称为内部总线；在 CPU 与内存 RAM、ROM 和 I/O 设备接口之间传输数据所用的总线称为外部总线。
	按总线功能，可以分为地址总线、数据总线、控制总线。
	按总线在危机系统中的位置，可以分为机内总线和机外总线。
	按总线用途，可以分为局部总线、系统总线和通信总线。
	按总线中数据线的多少，可以分为并行总线和串行总线。
	
	总线标准主要规定总线的机械结构规范、功能结构规范和电气规范。
	
	总线的主要性能包括 5 种：总线宽度、总线带宽、总线负载、总线分时复用、总线猝发传输。

1.3.3 I/O 接口

	I/O 接口（简称为接口）是为了解决主机和外设在信息形式、工作速度方面等差异而设置的，也称为 I/O 控制器。通过接口可以实现主机和外设之间的信息交换。
	● I/O 接口的功能、分类和常见的接口标准
	I/O 接口的功能包括：
	（1）实现主机与外设的通信联络控制。
	（2）进行地址译码和设备选择。
	（3）实现数据缓冲。
	（4）数据格式变换。
	（5）传递控制命令和状态信息。
	
	按 I/O 的信号分类，可分为数字接口和模拟接口。
	按通用性分类，可分为通用接口和专用接口。
	按灵活性分类，可分为可编程接口和不可编程接口。
	按交换数据方式分类，可分为串行接口和并行接口。串行通信又可分为同步通信和异步通信。
	
	常见的接口标准包括：IDE、ATA、SATA、eSATA、SCSI、PCMCIA、IEEE-1394、USB
	● I/O 端口
	I/O 端口（简称为端口）是指接口电路中可以被 CPU 直接访问的寄存器，由若干个端口加上相应的控制逻辑电路组成接口。CPU 通过输入指令可以从端口读取信息，通过输出指令可以把信息写入端口。

### 1.4 指令系统

1.4.2 复杂指令系统(CISC)

	● CISC指令系统的特点
		指令数量众多

## 6.项目管理

### 6.1 项目管理概述

	● 项目管理的定义、对象和目的
	项目是在特定条件下，具有特定目标的一次性任务，是在一定时间内，满足一系列特定目标的多项相关工作的总称。
	项目管理是管理学的一个分支学科，是指在项目活动中运用专门的知识、技能、工具和方法，使项目能够在有限资源限定条件下，实现或超过设定的需求和期望的过程。
	项目管理是对一些成功达成一系列目标的相关活动（譬如任务）的整体检测和管控，包括策划、进度计划和维护组成项目的活动的进展。

### 6.2 范围管理

6.2.1 范围计划的编制

	● 范围管理计划的内容
	范围管理计划是对项目的范围进行确定、记载、核实管理和控制的行动指南。与项目范围计划不同（范围计划描述的是项目的边界），范围管理计划描述的是项目组将如何进行项目的范围管理。

6.2.2 创建工作分解结构

	● WBS基本概念
	WBS, Work Breakdown Structure：工作分解结构，把项目整体或主要的可交付成果分解成容易管理、方便控制的若干个子项目，子项目需要继续分解为工作包，直到整个项目都分解为可管理的工作包。这些工作包的总和就是项目的所有工作范围。
	最底层的工作单元称为工作包，它应该便于完整地分派给不同的人或组织，所以要求明确各工作单元直接的界面。工作包应该非常具体，以便承担者能明确自己的任务、努力的目标和承担的责任。所有工作包的描述必需让成本会计管理者和项目监管人员理解，并能够清楚地区分不同工作包的工作。同时，工作包的大小也是需要考虑的细节，如果工作包太大，则难以达到可管理和可控制的目标；如果工作包太小，则WBS就要消耗项目团队成员大量时间和精力。

6.2.3 范围确认和控制

	● 范围确认
		范围确认主要是确认项目的可交付成果是否满足项目干系人的要求。项目干系人进行范围确认时，检查内容主要包括：
		（1）可交付成果是否是确定的、可核实的。
		（2）每个可交付成果是否有明确的里程碑，里程碑是否有明确的、可辨别的事件。
		（3）是否有明确的质量标准。
		（4）审核和承诺是否有清晰的表达。
		（5）项目范围是否覆盖了需要完成的产品（或服务）进行的所有活动，有没有遗漏或者错误。
		（6）项目范围的风险是否太高，管理层是否能够降低可预见的风险发生时对项目的冲击。
	● 范围控制
		项目范围定义了项目应该做的和不应该做的，那么对于范围变更，就不能随意进行。对于用户不断提出的新要求和建议，项目组应该坚持“决不让步，除非交换”的原则，尽可能减少范围蔓延的可能性，所有的范围变更必须在项目的工期、费用或者质量要求上得到相应的变更。
		所有的变更必须记载，范围控制必须能够对造成范围变更的因素施加影响，估算对项目的资金、进度和风险等的影响，保证变化是有利的。

### 6.3 进度管理

	进度管理就是采用科学的方法，确定进度目标，编制进度计划和资源供应计划，进行进度控制，在与质量、成本目标协调的基础上，实现工期目标。
	进度管理包括以下过程：
		活动定义
		活动排序
		活动资源估算
		活动历时估算

6.3.1 活动排序

	● 前导图法PDM
		Precedence Diagramming Method, 用放个或矩形（节点）表示活动，用箭线表示依赖关系。每个节点的活动都有最早开始时间ES、最迟开始时间LS、最早结束时间EF和最迟结束时间LF。
	● 箭线图法ADM
		Arrow Diagramming Method, 用节点表示事件，用箭线表示活动，并在节点处将其连接起来，表示依赖关系。活动的开始事件（键尾）叫做紧前事件，活动的结束事件（键头）叫做紧后事件。
	● 确定依赖关系
		强制性依赖关系
		可自由处理的依赖关系
		外部依赖关系

6.3.2 活动资源估算

		自下而上的估算

6.3.3 活动历时估算

	● 软件项目的工作量
		代码行LOC（Line of Code）
		累计投入工作量 * 单位工作量的成本 / 代码行 = 单位工作量的代码行产出。
	● 德尔菲法
		专家估算 a：最少代码行数 m：最可能的代码行数 b：最多代码行数，计算专家期望：(a + 4m + b)/6。根据专家数量加权平均。
	● 类比估算法
		评估与应用领域、环境和复杂度方面相似的历史项目，通过新项目与历史项目的比较得到规模估计。
	● 功能点估算法
		在需求分析阶段基于系统功能的规模估计方法。
	● COCOMO模型
		组织性、嵌入性和半独立型

6.3.4 进度控制

	● 分析进度偏差的影响
		当出现进度偏差时，需要分析该偏差对后续活动及总工期的影响：
		分析产生偏差的活动是否为关键活动
		分析进度偏差是否大于总时差
		分析进度偏差是否大于自由时差
	● 项目进度计划的调整
		关键活动的调整：
			关键活动的实际进度较计划进度提前
			关键活动的实际进度较计划进度落后
		非关键活动的调整：
			在总时差范围内演唱非关键活动的持续时间
			缩短工作的持续时间
			调整工作的开始或完成时间
		增减工作项目：
			增加工作项目
			减少工作项目
		资源调整：
			资源优化，提高资源利用率

6.4.2 成本预算

	● 直接成本与间接成本
		非直接成本
		隐没成本
		学习曲线：如果采用了项目成员未使用过的技术和方法，那么在使用这些技术和方法的初期，项目成员有一个学习的过程，很多时间和劳动投入到尝试和试验中。这些尝试和试验会增加项目的成本。

		质量要求

		为范围和成本的潜在变化而预留的预算。

6.4.3 成本控制

	● 成本控制基本概念
		项目成本控制是按照事先确定的成本基准计划，通过运用多种恰当的方法，对项目实施过程中所消耗费用的使用情况进行管理和控制，以确保项目的实际成本限制在项目成本预算范围内。
	● 挣值分析方法
	挣值分析通过测量和计算己完成工作的预算费用、已完成工作的实际费用和计划工作的预算费用，得到有关计划实施的进度和费用偏差，达到判断项目预算和进度计划执行情况的目的。
	计划工作量的预算成本PV = 计划工作量 × 预算定额
	已完成工作量的实际成本AC
	已完成工作量的预算成本EV = 已完成工作量 × 预算定额
	完工预算BAC = 完成时的PV总和
	进度偏差SV = EV - PV
	进度绩效指数SPI = EV/PV
	成本偏差CV = EV - AC
	成本绩效指数CPI = EV/AC
	剩余工作的成本ETC = BAC - EV（非典型偏差） 或 (BAC - EV)/CPI（典型偏差）
	完工估算EAC = AC + ETC

### 6.8 风险管理

6.8.1 风险管理的概念

	● 风险的定义和基本属性
	项目风险管理就是项目管理人员通过风险识别、风险估计和评价，并以此为基础合理地使用多种管理方法、技术和手段，对项目活动涉及的风险实行有效的控制。
	风险表达了一种概率，具有偶发性。项目风险可以简单地理解为项目中的不确定因素。，风险发生后可能给项目带来问题，也可能带来机会。
	风险具有两个基本属性：
		随机性：风险事件的发生及其后果都具有偶然性
		相对性：风险总是相对项目活动主体而言的，同样的风险对于不同的主体有不同的影响。

6.8.2 风险管理的过程

	● 风险管理计划编制
	风险管理计划描述的是如何安排与实施项目风险管理，它是项目开发计划的从属计划。
	● 风险识别
	风险识别就是采用系统化的方法，确定风险的来源、风险产生的条件，风险的特征和可能影响的范围。
	● 风险定性分析
	风险可能性与影响分析
	确定风险优先级
	确定风险类型
	● 风险定量分析
		灵敏度分析
		期望货币价值分析
		决策树分析
		蒙特卡洛模拟：找一组随机数作为统计模拟用，这一方法的精度在于随机数的均匀性和独立性。
	● 风险应对计划编制
	防范策略：风险发生前，采取一定的措施对风险进行防范。
	响应策略：风险发生后，采取相应措施，降低风险带来的损失。
	● 风险监控
	分总已识别的风险，监测参与风险和识别新的风险，保证风险计划的执行。

## 8. 系统规划与分析

### 8.1 系统规划概述

	● 系统规划的需求
	系统规划是信息系统生命周期的第一个阶段，其任务是对企业的环境、目标及现有系统的状况进行初步调查，根据企业目标和发展战略，确定信息系统的发展战略，对建设新系统的需求做出分析和预测，同时考虑建设新系统所受的各种约束，研宄建设新系统的必要性和可能性。根据需要与可能，给出拟建系统的备选方案。对这些方案进行可行性分析，写出可行性研宄报告。可行性研宄报告审议通过后，将新系统建设方案及实施计划编写成系统设计任务书。
	系统规划的需求问题主要体现在：
	系统搞建设与企业目标和发展战略不匹配
	系统建设周期长
	企业组织结构陈旧，管理思想落后，使信息系统沦为摆设
	企业人员素质较低，不能合理地操作和使用信息系统，更谈不上系统运行管理和维护
	技术方案不合理
	系统开发、运行与维护的标准、规范比较混乱
	资源短缺，投入太少，而对系统的期望过高
	● 系统规划的主要步骤

	分析子系统的组成和基本功能
	拟定系统的实施方案
	进行系统的可行性研究
	制定系统建设方案

### 8.2 项目的提出与选择

8.2.1 项目的立项目标和动机

	● 进行基础研究
	此类项目通常由大学、科研院所、企业集团从事基础研究的部门提出和实施。
	基础研究更多体现为一种探索性研究，成果多体现为某种理论体系和技术成果，通常没有具体的产品发布目标，也没有苛刻的时间限制。
	● 进行应用研发
	此类项目通常由企业进行立项和开发。
	企业立项的基本动机时得到应用产品，并向目标客户群进行销售，从而占有市场份额并获取利润。
	● 提供技术服务
	对此类项目进行立项的企业通常能像目标客户提供比较全面的技术服务，而不是单一的软件产品。
	● 产品的使用者
	产品的使用者是系统的最终用户。对他们来说，项目立项的动机是通过采购产品或技术服务来得到使用价值。

8.2.2 项目立项的价值判断

	● 价值判断方法
	通常，以基础研究为目标的项目是高度技术研宄导向的，以应用产品开发为目标的项目重点关注的是技术在具体领域中的应用和推广，以技术服务为目标的项目则是高度客户业务导向或客户满意导向的，产品的最终用户则主要关注系统的使用、影响和代价等应用性问题。
	企业并不把系统立项和开发完成看作是获取价值的终点，而仅仅是一个起点。
	企业最终需要的可能是获取利润、占有市场份额、提高影响力、广泛的社会效益等这些潜在的商业价值目标，信息系统则常被作为支持性手段来支撑这些目标的实现。

8.2.3 项目的选择和确定

	● 选择有核心价值的项目
	有核心价值的项目总是和企业的核心业务相关的，也可以说，信息化的关键就是核心业务的信息化。
	● 评估所选择的项目在判断出一个具有潜在价值的项目后，还应评估项目实施的约束、风险、成本和效益。
	项目约束，是指在系统开发过程中，“不能做什么”的原则。这些约束有些来自客户，有些来自企业本身，还有些来自外部环境，可能包括企业约束、资源约束、能力约束、环境约束和用户约束等。
	● 项目优先级排序
	根据企业已有资源情况，进行项目的成本效益分析，考察净现值、投资回收期等指标。
	● 评估项目的多种实施方式
	企业开发信息系统，既可以自己组建开发团队进行项目开发，也可以把系统开发任务承包给其他企业，或者购买产品并进行系统集成，还可以自己完成技术方案和设计，然后把编码和测试任务进行外包等。
	对这些项目实施方式的取舍，主要依据是对项目风险、收益和资源开销等方面的考虑，其根本目标是为了优化和合理运用投入项目的资源。
	● 平衡地选择合适的方案
	技术风险
	用户锁定行
	扩展性
	目标偏离
	事实上，对系统功能和性能的要求常常是充满矛盾的，任何时候都不存在一个完美无缺的方案，只存在一个对当前的项目目标相对比较合适的方案。选择项目的基本原则应该是“合适”，而不是尽可能的“好”。实际上，任何超出预期设定目标的“好”性能，通常都意味着某方面更多的成本或者潜在的风险。

### 8.3 系统分析概述

	● 系统分析的任务
	系统分析阶段的基本任务是系统分析师和用户在充分了解用户需求的基础上，把双方对新系统的理解表达为系统需求规格说明书。
	● 系统分析的难点
	开发软件系统最困难的部分就是准确说明开发什么，因为用户往往很难给出完整、正确的原始需求，也很难想象出未来的软件应该提供哪些功能，以解决自己的业务问题。这些都需要软件开发人员协助，通过多次的讨论方能最终确认。
	难点主要包括以下三个方面：
	系统该分析师和用户对系统的理解不同。
	系统分析师和用户沟通困难。
	环境的不断变化。

### 8.4 问题分析

	系统问题分析是软件工程中的一个重要阶段，它是指通过对问题领域进行综合分析，以识别和定义系统问题，从而为后续的软件开发和系统设计提供指导和支持。

8.4.1 研究问题的领域

	● 上下文图
	上下文图是一种用于显示系统或过程在其环境中与其他实体之间的交互作用的图形工具。上下文图显示系统或过程与外部实体之间的输入和输出关系，以及它们之间的控制和反馈关系。
	构建上下文图包括以下几个步骤：
	确定系统的边界。
	识别外部实体。
	识别过程。
	绘制上下文图。
	验证上下文。

8.4.2 分析问题和机会

	● 因果分析
	在陈述任何可能的问题之前真正地分析这个问题，分析每个问题的原因和结果。结果可能是一个不同的、更深的或更基本的问题的症状。对于那个问题也必须分析原因和结果，直到原因和结果不再引发其他问题。因果分析可以得出对问题的真正理解，并且能够得出虽然并不十分明显但更有创造性和更有价值的方案。
	问题、机会、目标和约束矩阵。

8.4.3 制定系统改进目标

### 8.5 业务流程分析

8.5.1 业务流程分析概述

	组织结构图描述了系统内部各部门的划分，以及这些部门之间的相互关系，功能分析图则反映了这些部门所具有的管理功能。下一步的任务就是要明确这些职能是如何在有关部门具体完成的，以及在完成这些职能时信息处理工作的一些细节情况，这项工作称为业务流程分析。
	业务流程分析的目的是了解各个业务流程的过程，明确各个部门之间的业务关系和每个业务处理的意义，为业务流程的合理化改造提供建议，为系统的数据流程变化提供依据。
	企业业务流程的整体目标是为顾客创造价值，因此，以顾客利益为中心，以员工为中心，以及以效率和效益为中心是业务流程的核心。
	信息系统是管理创新，它的运行基础是企业的业务流程。业务流程不通畅是导致企业信息系统项目失败的主要原因之一。
	
	● 业务流程分析的步骤
	业务流程分析的主要任务是调查系统中各环节的业务活动，掌握业务的内容和作用，以及信息的输入、输出、数据存储和信息处理方法及过程等。
	具体步骤如下：
	调查
	描述现有业务流程
	确认现有业务流程
	分析
	发现问题并提出解决方案
	● 业务流程分析的方法
	价值链分析法：运用系统性方法来考察企业各项活动和项目关系，从而找寻具有竞争优势的资源。价值链就是创造价值的工作流程，可以把企业具体的活动细分为：生产指挥流程、计划决策流程、营销流程、信息搜集与控制流程、资金筹措流程等。
	客户关系分析法：处理有关客户的数据，分析他们与企业的关系，以提高企业未来的销售、服务和成本控制的过程。
	供应链分析法：从企业供应链的角度分析企业的业务流程。
	基于ERP的分析法：将企业业务流程看作一个紧密连接的供应链，对供应链上的所有环节进行有效管理。
	业务流程重组：通过重新审视企业的价值链，从功能成本的比较分析中，确定企业在那些环节具有比较优势。
	● 业务流程分析的工具
	业务流程图TFD（Transaction Flow Diagram）
	业务活动图BAM（Business Activity Mapping）
	统一建模语言UML（Unified Modeling Language）

8.7.1 可行性评价准则

	经济可行性也称为投资收益分析或成本效益分析，主要评估项目的建设成本、运行成本和项目建成后可能的经济收益。

	技术
	资源
	目标
	● 法律可行性
	法律可行性也称为社会可行性，具有比较广泛的内容，它需要从政策、法律、道德、制度等社会因素来论证信息系统建设的现实性。
	● 用户使用可行性
	用户使用可行性也称为执行可行性，是从信息系统用户的角度来评估系统的可行性，包括企业的行政管理和工作制度、使用人员的素质和培训要求等，可以细分为：
	管理可行性，分析从企业管理上建设可行性。
	运行可行性，分析和测定信息系统在确定环境中能够有效工作。

	● 净现值分析
	净现值NPV（Net Present Value），项目生命周期内各年的现金流量按照相同的折现率折现到现值之和。
	● 项目案例分析
	● 净现值率
	净现值率NPVR（Net Present Value Rate） = 净现值NPV / 项目总投资额现值P

8.8.3 投资回收期与投资回报率

	● 静态投资回收期
	静态投资回收期 = 累计净现金流量开始出现正值的年份数 - 1 + |上年累计净现金流量|/当年净现金流量
	● 动态投资回收期
	动态投资回收期 = 累计折现值开始出现正值的年份数 - 1 + |上年累计折现值|/当年折现值
	● 投资回收率
	投资回收率 = 1 / 动态投资回收期
	● 投资收益率
	投资收益率 = 投资收益 / 投资成本

## 9.软件需求工程

### 9.1 软件需求概述

9.1.1 需求的层次

	软件需求是指用户对新系统在功能、行为、性能、设计约束等方面的期望。
	软件需求时多层次的，包括业务需求、用户需求和系统需求：
	● 业务需求
	业务需求是指反映企业或客户对系统高层次的目标要求，通常来自项目投资人、购买产品的客户、客户单位的管理人员、市场营销部门或产品策划部门等。
	● 用户需求
	用户需求描述的时用户的具体目标，或用户要求系统必须能完成的任务。
	通常采取用户访谈和问卷调查等方式，对用户使用的场景进行整理，从而建立用户需求。
	● 系统需求
	系统需求是从系统的角度来说明软件的需求，包括功能需求、非功能需求和设计约束等。

9.1.2 质量功能部署QFD

	质量功能部署QFD（Quality Function Deployment）是一种将用户要求转化成软件需求的技术，其目的是最大限度地提升软件工程过程中用户的满意度。
	● 常规需求
	用户认为系统应该做到的功能或性能，实现越多用户会越满意。
	● 期望需求
	用户想当然认为系统应具备的功能或性能，但并不能正确描述自己想要得到的这些功能或性能需求。如果期望需求没有得到实现，用户会感到不满意。
	● 意外需求
	也称兴奋需求，用户要求范围外的功能或性能，实现这些需求用户会更高兴，但不是先也不影响其购买的决策。

9.1.3 软件需求

	● 软件需求定义
	软件需求目前并没有统一的定义，一般包括以下几个方面的内容：
	（1）用户解决问题或达到目标所需的条件或权能。
	（2）满足合同、标准、规范或其他正式规定文档所需具有的条件或权能。
	（3）一种反映（1）或（2）所述条件或权能的文档说明，即需求规格说明。
	● FURPS+模型
	（1）功能性Functional
	（2）可用性Usability
	（3）可靠性Reliability
	（4）性能Performance
	（5）可支持Sptability
	IEEE在软件需求规格说明标准中，把好的需求定义为正确、无歧义、可检验和可跟踪的需求。把好的需求集合定义为完整、一致和可修改的需求集合。把高品质软件需求特性定义为正确性、完整性、一致性、明确性、可验证性、依照重要程度及稳定度进行优先排序、可修改性、可追溯性、可理解性。

9.1.4 需求工程

	● 需求获取
	通过与用户的交流，对现有系统的观察，对任务的分析，获取需求。
	● 需求分析
	● 形成需求规格（需求文档化）
	● 需求确认与验证
	以需求规格说明为输入，通过用户确认、复审会议、符号执行、模拟仿真或快速原型等途径与方法，确认和验证需求规格的完整性、正确性、一致性、可测试性和可行性，包含有效性检查、一致性检查、可行性检查和确认可验证。
	● 需求管理
	需求管理包括需求文档的追踪管理、变更控制、版本控制等管理性活动。

### 9.2 需求获取

9.2.1 用户访谈

	需求获取是一个确定和理解不同的项目干系人的需求和约束的过程。需求获取是一件看上去很简单，做起来却很难的事情。需求获取是否科学、准确、充分，对获取的结果影响很大，这是因为大部分用户无法完整地描述需求，而且也不可能看到系统的全貌。
	系统分析师必须建立一个对问题进行彻底探讨的环境，而这些问题与将要开发的系统有关，让用户明确了解，对于某些功能的讨论并不意味着即将在系统中实现它。
	● 准备访谈
	结构化形式、非结构化形式。
	● 主持访谈
	限制访谈时间、寻找异常和错误情况、深入调查细节、认真做好记录。
	● 访谈的后续工作
	吸收、理解和记录访谈所获得的信息。

9.2.2 问卷调查

	● 问卷调查的优点
	在短时间内，以低廉的代价从大量的回答中收集数据。
	● 问卷调查的不足
	缺乏灵活性。

9.2.3 采样

	● 采样的定义
	从种群中系统地选出有代表性样本集，通过认真研究所选出的样本集，可以从整体上揭示种群的有用信息。
	● 采样的分类
	随机采样、分层采样、聚类采样、系统采样。
	● 采样技术的优点
	加快了数据收集的过程，提高了效率，减少数据收集的偏差。
	● 采样技术的不足
	样本规模的确定依赖于期望的可信度和已有的先验知识，很大程度上取决于系统分析师的主观因素。

9.2.4 情节串联板

	● 情节串联板的概念
	通常是一系列图片，系统分析师通过这些图片来讲故事。
	● 情节串联板的类型
	被动式
	主动式
	交互式
	● 情节串联板的优缺点
	给用户一个直观的演示，用户友好，交互性强。
	花费时间很多，需求获取速度降低。

9.2.5 联合需求计划JRP

	联合需求计划JRP（Joint Requirement Planning）是一个通过高度组织的群体会议来分析企业内的问题并获取需求的过程，它是联合应用开发JAD（Joint Application Development）的一部分。
	● 联合应用开发JAD
	JAD是以小组形式定义和建立系统，是一个专题讨论组，由这个专题讨论组来定义并详细说明系统的需求和可选的技术方案。
	● JRP会议
	● 主要原则
	JPR的主要意图是收集需求，而不是对需求进行分析和验证。

9.2.6 需求记录技术

	● 任务卡片
	任务、目的、触发、前提、频率、关键情况、子任务、任务变体
	● 场景说明
	场景说明就是用户对其工作场景和过程的详细描述。
	● 用户故事
	用户故事描述了对用户有价值的功能，分别是书面描述、交谈和测试用例。
	● Volere白卡
	Volere白卡是一种类似于任务卡片的需求记录工具。
	用户故事和Volere比卡定位的是最小的需求项，一般在敏捷方法中使用。

	● PDOA方法

### 9.4 结构化分析方法SA

	SA方法的基本思想是自顶向下，逐层分解，把一个大问题分解成若干个小问题，每个小问题再分解。经过逐层分解，每个最底层的问题都是足够简单、容易解决的。
	SA方法的核心是数据字典，围绕着数据字典，有功能模型、行为模型和数据模型。

9.4.1 数据流图DFD

	● DFD的主要作用

	● DFD的基本符号

	加工

	外部实体
	● DFD的层次
	顶层图
	逐层分解
	● 如何画DFD
	画出系统的输入和输出
	画出DFD的内部
	为每一个数据流命名
	为加工命名

9.4.2 状态转换图STD

	● STD的定义
	描述系统的状态和引起系统状态转换的事件。
	● STD的基本符号
	圆形或椭圆形：状态
	箭头线：状态转换
	实心圆：初始状态/最终状态
	● STD的作用
	既可以表示循环运行的过程，也可以表示单程生命周期。

9.4.3 数据字典

	数据字典是在DFD的基础上，对DFD中出现的所有命名元素都加以定义。
	● 数据字典的条目

### 9.6 需求定义

9.6.1 需求定义的方法

	● 严格定义方法
	需求的严格定义建立在以下的基本假设之上：
	所有需求都能够被预先定义。
	开发人员与用户之间能够准确而清晰地交流。
	采用图形（或文字）可以充分体现最终系统。
	● 原型方法
	原型化的需求定义过程是一个开发人员与用户通力合作的反复过程。从一个能满足用户基本需求的原型系统开始，允许用户在开发过程中提出更好的要求，根据用户的要求不断地对系统进行完善，它本质上是一种迭代的循环型开发方式。
	采用原型方法需要注意以下几个问题：
	并非所有的需求都能在系统开发前被准确地说明。
	项目干系人之间通常都存在交流上的困难，原型提供了克服该困难的一个手段。
	需要实际的、可供用户参与的系统模型。
	有合适的系统开发环境。
	反复是完全需要和值得提倡的。

## 10.软件架构设计

### 10.1 软件架构概述

	在软件需求和设计之间存在一条很难逾越的鸿沟，从而很难有效地将需求转换为相应的设计。为此，学者们提出了软件架构（Software Architecture）的概念，并试图在软件需求与设计之间架起一座桥梁，解决系统结构和需求实现平坦过度的问题。
	软件架构为软件系统提供了一个结构、行为和属性的高级抽象，由构件的描述、构件的相互作用（连接件）、指导构件集成的模式，以及这些模式的约束构成。
	软件架构指定了系统的组织结构和拓扑结构，并且显示了系统需求和构件之间的对应关系，提供了一些设计决策的基本原理。
	软件架构己完全独立于软件工程，成为计算机科学的一个最新的研宄方向和独立学科分支。
	
	● 软件架构的意义
	架构是项目干系人进行交流的手段。
	架构是早期设计决策的体现。
	架构明确了对系统实现的约束条件。
	架构决定了开发和维护组织的组织结构。
	架构制约着系统的质量属性。
	架构使推理和控制更改更简单。
	架构有助于循序渐进的原型设计。
	架构可以作为培训的基础。
	架构是可传递和可复用的模型。
	● 软件架构的发展史
	无架构设计阶段
	萌芽阶段
	初级阶段
	高级阶段

### 10.3 软件架构风格

10.3.1 软件架构风格概述

	● 软件架构风格的概念
	软件架构风格是描述某一特定应用领域中系统组织方式的惯用模式。

10.3.2 数据流体系结构风格

	数据流体系结构是一种计算机体系结构，直接与传统的冯诺依曼结构或控制流体系结构进行了对比。
	● 批处理体系结构风格
	在批处理风格的软件体系结构中，每个处理步骤是一个单独的程序，每一步必须在前一步结束后才能开始，并且数据必须是完整的，以整体的方式传递。
	● 管道-过滤器体系结构风格
	管道-过滤器体系结构风格的基本构件是过滤器，连接件是数据流传输管道，将一个过滤器的输出传到另一个过滤器的输入。

10.3.3 调用/返回体系结构风格

	● 主程序/子程序风格
	主程序/子程序风格一般采用单线程控制，把问题划分为若干处理步骤，构件即为主程序和子程序。
	● 面向对象体系结构风格
	数据的表示方法和他们的相应操作封装在一个抽象数据类型或对象中。
	● 层次型体系结构风格
	每一层为上层提供服务，并作为下层的客户。
	● 客户端/服务器体系结构风格
	二层C/S架构
	三层C/S架构：表示层、功能层（业务逻辑层）、数据层
	B/S架构：是三层C/S架构的一种实现方式


10.3.4 以数据为中心的体系结构风格

	● 仓库体系结构风格
	中央数据结构（当前数据的状态）
	对中央数据进行操作的独立构件
	● 黑板体系结构风格
	运用多种不同的知识源，使得问题的表达、组织和求解变得比较容易。
	黑板系统是一种问题求解模型，是组织推理步骤、控制状态数据和问题求解的领域知识的概念框架。

10.3.5 虚拟机体系结构风格

	● 解释器体系结构风格
	一个解释器通常啊包括：完成解释工作的解释引擎、一个包含将被解释的代码的存储区、一个记录解释引擎当前工作状态的数据结构、一个记录有源代码被解释执行进度的数据结构。
	● 规则系统体系结构风格
	规则集、规则解释器、规则/数据选择器、工作内存。

10.3.6 独立构件体系结构风格

	● 进程通信体系结构风格
	构件是独立的过程，连接件是消息传递。
	● 事件系统体系结构风格
	基于实践的隐士调用风格，构件不直接调用一个过程，而是出发或广播一个或多个事件。

10.3.7 面向服务的架构风格

	● 面向服务的架构 (SOA) 概述
	SOA是一种在计算环境中设计、开发、部署和管理离散逻辑单元（服务）模型的方法。
	W3C的定义：SOA是一种应用程序架构，在这种架构中，所有功能都定义为独立的服务，这些服务带有明确的可调用结构，能够以定义好的顺序调用这些服务来形成业务流程。
	Service-architecture.com的定义：服务是精确定义、封装完善、独立于其他服务所处环境和状态的函数。
	Gartner的定义：SOA是一种C/S架构的软件设计方法，应由服务和服务使用者组成，SOA着重强调构件的松散耦合，并使用独立的标准接口。
	● SOA的关键技术
	与SOA紧密相关的技术主要有：统一描述、发现与集成UDDI（Universal Description Discovery and Integration）、Web服务描述语言WSDL（Web Service Description Language）、表属性状态转移REST（Respresentational State Transfer），这些技术都是以XML为基础发展起来的。
	● SOA的实现方法
	Web Service：服务提供者、服务注册中心、服务请求者
	服务注册表
	企业服务总线

10.6.3 质量属性场景描述

	● 质量属性场景组成
	质量属性场景是一个具体的质量属性需求，是利益相关者与系统的交互的简短陈述：
	刺激源：某个生成该刺激的实体
	刺激：刺激到达系统时需要考虑的条件
	环境：刺激在某些条件下发生
	制品：某个制品被激励，可能时整个系统，也可能时系统的一部分
	响应：刺激到达后所采取的行动
	响应度量：响应发生时，应当能以某种方式对其进行度量
	● 可用性质量属性场景
	可用性质量属性场景所关注的方面包括系统故障发生的频率、出现故障时会发生什么情况、允许系统有多长时间正常响应、什么时候可以安全地出现故障、如何防止故障的发生以及发生故障时要求进行哪种通知。
	刺激源：系统内部、系统外部
	刺激：疏忽、错误、崩溃、时间
	环境：正常操作、降级模式
	制品：系统处理器、通信信道、持久存储器、进程
	响应：系统应检测事件
	响应度量：系统必须可用的时间间隔、可用时间、系统可以在降级模式下运行的时间间隔、故障修复时间
	● 可修改性质量属性场景
	可修改性质量属性场景主要关注系统在改变功能、质量属性时需要付出的成本和难度。
	可能发生在系统设计、编译、构件、运行等多种情况和环境下。
	● 性能质量属性场景
	性能质量属性场景主要关注系统的响应速度，可以 通过效率、响应时间、吞吐量、负载来客观评价性能的好坏。
	● 可测试性质量属性场景
	可测试性质量属性场景主要关注系统测试过程中的效率，发现系统缺陷或故障的难易程度。
	● 易用性质量属性场景
	易用性质量场景主要关注用户在使用系统时的容易程度，包括系统的学习曲线、完成操作的效率、对系统使用过程的满意程度等。
	● 安全性质量属性场景
	安全性质量属性场景主要关注系统在安全性方面的要素，衡量系统在向合法用户提供服务的同时，组织非授权用户使用的能力。

## 11.系统设计

### 11.1 系统设计概述

	系统设计的主要内容包括概要设计和详细设计。
	概要设计又称为系统总体结构设计。其主要任务是将系统的功能需求分配给软件模块，确定每个模块的功能和调用关系，形成软件的模块结构图，即系统结构图。
	详细设计为每个具体任务选择适当的技术手段和处理方法。
	
	● 网络设计
	网络设计的主要任务是：根据系统的要求选择网络结构，按照系统结构的划分，安排网络和设备的分布。
	● 代码设计
	代码设计的主要目的是确保代码的唯一化、规范化和系统化。
	● 输入/输出设计
	输入设计的目的是确保向系统输入数据的完整性、正确性和一致性。
	输入设计的原则：输入数据最少原则、简单性原则、尽早验证原则、少转换原则。
	输出设计的目的是确保系统输出数据的完整性、正确性和一致性。
	● 处理流程设计
	处理流程设计的主要目的是确定各个系统模块的内部结构，包括局部数据组织和控制流，以及每个具体加工过程和实施细节。
	● 数据存储设计
	数据存储设计的主要目的是根据数据处理要求、处理方式、存储的信息量大小、数据使用的频率和所提供的设备条件等，选择数据存储的方式、存储介质、数据组织方式和记录格式，并估算数据的容量。
	● 用户界面设计
	界面是系统与用户交互的最直接的层面，界面的好坏决定用户对系统的第一印象，而设计优良的界面能够引导用户自己完成相应的操作，起到向导的作用。
	良好的用户界面设计需要遵循以下原则：置于用户控制之下、减轻用户的记忆负担、保持界面一致性。通常称之为人机交互的黄金三原则。
	● 安全性和可靠性设计
	确保系统的安全性和可靠性，对系统的运行环境和数据处理进行有效控制。

### 11.2 处理流程设计

11.2.1 流程设计概述

	系统处理流程对应于现实世界中的真实业务过程，通过对业务流程的设计，可以对其进行建模，以便使用信息系统来取代传统的手工处理，提高业务处理的效率和准确性，降低业务处理成本
	
	● 流程
	业务流程：一组将输入转化为输出的相互关联或相互作用的活动。
	流程包含6个基本要素，分别是输入资源、活动、活动的相互作用、输出结果、用户和价值。
	● 工作流
	工作流：一类能够完全或部分自动执行的业务过程。根据一系列过程规则、文档、信息或任务，在不同的执行者之间传递和执行。
	● 活动及其所有者
	活动是流程定义中的一个基本要素，一次活动可以改变流程处理数据的内容、流程的状态，并可能将流程推动到其他活动中去。
	活动的所有者是流程参与者之一，他们有权决定该活动是否结束。
	● 工作项
	工作项代表流程实例种活动的参与者将要执行的工作。


11.2.2 工作流管理系统WFMS

	工作流管理系统WFMS（WorkFlow Management System）通过软件定义、创建工作流，并管理其执行。
	● WFMS的基本功能
	对工作流进行建模：即定义工作流
	工作流执行
	业务过程的管理和分析
	● WFMS的组成
	工作流执行服务：是WFMS的核心模块，功能包括创建和管理流程定义，创建、管理和执行流程实例。
	工作流引擎：为流程实例提供运行环境。
	流程定义工具：管理流程定义的工具，通过图形方式把复杂的流程定义显示出来并加以操作。
	客户端应用：通过请求的方式与工作流执行服务交互。
	调用应用：被工作流执行服务调用的应用，调用应用与工作流执行服务交互。
	管理监控工具：流行执行情况的监控，参与者等数据的维护管理。

11.2.3 流程设计工具

### 11.3 结构化设计

11.3.1 模块结构

	● 信息隐蔽与抽象
	信息隐蔽原则要求采用封装技术，将程序模块的实现细节隐藏起来，对于不需要这些信息的其他模块来说是不能访问的，使模块接口尽量简单。
	抽象原则要求抽取事务最基本的特性和行为，忽略非本质细节，采用分层次抽象的方式控制软件开发过程的复杂性。
	● 模块化
	模块是实现功能的基本单位。一般具有功能、逻辑、状态三个基本属性。
	当模块的大小使其实现代码行数约50-200行时，易于实现和维护。
	当模块的扇入和扇出系数为3或4，不应该超过7，否则会增大出错的概率。
	● 耦合
	耦合表示模块之间联系的程度。耦合度从底到高排序：
	非直接耦合：没有直接关系，通过主模块的控制和调用实现
	数据耦合：借助参数表传递简单数据
	标记耦合：借助参数表传递记录信息（数据结构）
	控制耦合：包含用于控制模块内部逻辑信息
	外部耦合：访问同一全局变量，而不是通过参数表
	公共耦合：访问统一公共数据
	内部耦合：访问另一个模块的内部数据
	● 内聚
	内聚表示模块内部各成分之间的联系程度。内聚度从高到低排序：
	功能内聚：完成一个单一功能，各部分协同工作
	顺序内聚：处理元素，顺序执行
	通信内聚：处理元素在一个数据结构上
	过程内聚：处理元素，按次序执行
	瞬时内聚：在同一时间间隔内执行
	逻辑内聚：完成逻辑相关的一组任务
	偶然内聚：完成一组没有关系或松散关系的任务
	● 模块类型
	传入模块
	传出模块
	变换模块
	协调模块

11.3.2 系统结构图

### 11.4 面向对象设计

	OOD是OOA方法的延续，其基本思想包括抽象、封装和可扩展性。

11.4.1 设计软件类

	类封装了信息和行为，是面向对象的重要组成部分，是具有相同属性、方法和关系的对象集合的总称。
	
	● 实体类
	实体类映射需求中的每个实体。
	实体类通常都是永久性的，他们所具有的属性和关系是长期需要的。
	● 控制类
	控制类是用于控制用例工作的类，一般是由动宾短语转换来的名词。
	通常情况下，控制类没有属性，只有方法。
	● 边界类
	边界类用于封装在用例内、外流动的信息或数据流。
	每个参与者和用例交互至少需要一个边界类，边界类使参与者能与系统交互。

11.4.2 对象持久化与数据库

	对象持久化（Persistence）是把内存中的对象保存到数据库或可永久保存的存储设备中。
	在多层软件设计和开发中，为了降低系统耦合度，一般会引入持久层。
	将对象持久化到关系数据库中，需要进行对象/关系映射ORM。
	
	● 容器管理持久化CMP
	CMP是由EJB容器来管理实体EJB的持久化。
	● Hibernate
	● iBatis
	● Java数据对象JDO

11.4.3 面向对象设计的原则

	● 开闭原则
	软件实体应对扩展开放，而对修改关闭，即尽量在不修改原有代码的情况下进行扩展。
	● 里氏替换原则
	一个软件实体如果使用的是一个基类对象，那么一定适用于其子类对象，而且察觉不出基类对象。即把所有基类都替换成它的子类，程序的行为没有变化。
	● 依赖倒置原则
	抽象不应该依赖于细节，细节应当依赖于抽象。即针对接口编程，而不是针对实现编程。
	● 组合/聚合复用原则
	在一个新的对象中，通过组合关系或聚合关系来使用一些有已有的对象，使之称为新对象的一部分，新对象通过委派调用已有对象的方法达到服用其已有功能的目的。
	● 接口隔离原则
	使用多个专门的接口，而不使用单一的总接口。每个接口应该承担一种相对独立的角色。
	● 最少知识原则
	也称为迪米特法则，是指一个软件实体应当尽可能少地与其他实体发生下相互作用。

### 12.2 软件测试概述

12.2.1 软件测试的概念

	寻找软件的缺陷及漏洞，通过测试提高程序的价值、可靠性或质量。
	IEEE对软件测试的定义：使用人工或自动的手段来运行或测量软件系统的过程，目的是检验 软件系统是否满足规定的需求，并找出与预期结果之间的差异。

12.2.2 软件测试的基本问题

	软件测试的基本问题是验证功能的实现和可用性，检查程序的错误。最终目的是提高用户体验，以较少的人力、物力和事件找出软件中潜在的各种错误和缺陷，通过修正错误和缺陷，回避软件发布后带来的商业风险。

12.2.3 软件测试的对象及目的

	源代码/目标代码、开发各阶段的文档配置数据

12.2.4 软件测试的原则

	测试用例的一个必要部分是预期的输出和结果的定义。用例必须包括输入数据和输入数据下程序正确输出的准确描述。
	程序员应该避免测试他们自己的程序。
	编写软件的组织不应当测试自己编写的软件。
	应当具有足够的耐心及细心检查每一个测试结果。
	测试用例不仅应给予有效和预期的输入，还应给予通常不会执行的无效和意外输入。
	应避免重新设计测试用例。
	计划测试工作不应默认假定不会发现错误。
	测试不要穷举。
	软件测试要尽早执行。
	软件测试应追溯到需求。

### 12.3 软件测试方法

12.3.1 按照被测程序是否可见分类

	● 黑盒测试
	又称为功能测试、数据驱动测试或基于需求规格说明书的功能测试。注重测试软件的功能性需求。黑河测试主要是为了发现以下错误：
	是否有功能错误，是否有功能遗漏。
	是否能够正确地接收输入数据并产生正确的输出结果。
	是否有数据结构错误或外部信息访问错误。
	是否有程序初始化和终止方面的错误。
	用例设计方法包括：等价类划分法、边界值分析法、错误推推测法、因果图法、正交试验设计法、功能图法和场景法。
	● 白盒测试
	又称为结构测试、透明盒测试、逻辑驱动测试或基于代码的测试。注重测试程序内部逻辑结构。
	用例设计方法包括：代码检查法、静态结构分析法、静态质量度量法、逻辑覆盖法、基本路径测试法、域测试、符号测试、路径覆盖和程序变异。
	覆盖标准包括：逻辑覆盖、循环覆盖和基本路径测试。
	● 灰盒测试
	介于白盒测试和黑盒测试之间的一种测试，不仅关注输入、输出的正确性，也关注程序内部的情况。
	通常是通过一些表征性的现象、事件、标志来判断内部的运行状态。

12.3.2 按照是否需要执行被测程序分类

	● 静态测试
	不运行被测试程序本身，仅通过分析或检查源代码的语法、结构、过程、接口等来检查程序的正确性。
	静态测试包括代码检查、静态结构分析、代码质量度量等。
	● 动态测试
	通过运行被测试程序，检查运行结果与预期结果的差异，并分析运行效率、正确性和健壮性等性能。

12.3.3 自动化测试

	● 自动化测试方法
	软件测试的自动化，在预设状态下运行应用从程序或者系统，评估运行结果。
	自动化测试是将人为驱动的测试行为转化为机器执行的过程。
	单元自动化测试、接口自动化测试、UI自动化测试。

### 12.4 软件测试类型

12.4.1 按测试对象划分

	● 功能测试
	功能测试根据产品特性、操作描述和用户场景来测试产品的特征和可操作行为，以确定其满足设计需求。
	功能测试又称为数据驱动测试，只需要考艾吕测试的各种功能，不需要考虑整个软件的内部结构和代码。
	● 性能测试
	性能测试是测试在不同情况下软件响应的时间，检查系统是否满足需求规格说明书中规定的性能。性能测试通过自动化测试工具模拟多种正常、峰值以及异常负载条件来对系统的各项性能指标进行测试。
	性能测试包括负载测试、压力测试、并发测试、容量测试、可靠性测试灯。
	● 安全测试
	安全测试是验证应用程序的安全等级和识别潜在的安全缺陷。主要是测试应用程序层的安全，包括应用程序本身的安全和数据安全。
	● 兼容性测试
	测试软件之间是否很好地运行。
	● 界面测试
	界面测试主要测试用户界面的功能模式布局是否合理、整体风格是否一直、各个控件的放置位置是否符合客户使用习惯，测试界面操作的便捷性、界面元素的可用性。
	● 易用性测试
	用户体验测试，交互的适应性、功能性和有效性。
	● 稳定性测试
	通过长时间运行软件来测试软件是否会出现问题。

## 13. 系统运行与维护

### 13.1 运维技术指标

	● 平均维修时间(MTTR)
	系统可修复故障的平均维修时间，即在给定时间段内发生的一次或多次系统故障的累计维修时间相对于故障次数的平均值。
	● 平均故障间隔时间(MTBF)
	系统两次故障之间的平均运行时间，即在给定周期内，系统发生一次或以上的故障之间累计运行时间相对故障次数的平均值。
	● 平均无故障时间(MTTF)
	可更换啊或不可修理系统或部件提供正常功能的平均持续时间。
	● 平均应答时间(MTTA)
	运维团队响应业务部门投诉、业务中断或异常事件所花费的平均时间。

## 17.数学与工程基础

	● 数学统计基础
	● 图论应用
		最小生成树：在连通的带权图的所有生成树中，权值和最小的生成树（所有顶点接入且没有回路）称作最小生成树。算法包括普利姆算法（Prim）、克鲁斯卡尔算法（Kruskal）、博鲁夫卡算法（Boruvka）和反向删除算法（Reverse-Delete）。
			普利姆算法：贪心算法，从一个顶点出发，寻找最近的顶点（最小权值的边）
			克鲁斯卡尔算法：贪心算法，排序逐步寻找局部最小权值的边，达到全局最小权值
	
		最短路径：带权图的最短路径问题即求两个顶点间路径上权值和最短的路径。算法包括迪杰斯特拉算法（Dijkstra）、弗洛伊德算法（Folyd）、贝尔曼-福特算法（Bellman-Ford）、A*算法（A-Star）、约翰逊算法（Johnson）、SPFA算法（Shortest Path Faster Alorithm）
			迪杰斯特拉算法：从起点出发，记录相邻最近的顶点的距离，重复
	
		网络与最大流量
			在网络图（可能非双向连接）中计算最大流量（瓶颈）
